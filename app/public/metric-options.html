Metric design

How should we score a model across many outputs?

Each model produces predictions for several tax and benefit outputs per household. To rank models we have to combine those into a single number. This page builds up the choices step by step on a tiny worked example, ending with the recommendation: data-derived global variable weights instead of the arbitrary 30% floor.

1The setup

Three households, five outputs (four dollar amounts plus one eligibility flag), two models. Every step on this page uses the same numbers.

Reference values (what PolicyEngine computes)

Dollar cells are signed amounts for one tax year — negatives are liabilities (taxes paid), positives are benefits received. The Medicaid column is a boolean eligibility flag; the paired dollar value shown below "Eligible" is PolicyEngine's per-capita Medicaid value for that household. We use it only for weighting (it never enters the LLM's prediction task).

Each household also has a market income (earnings, investment income, etc.) that is given to the model as input rather than predicted. It enters the weighting formulas as part of household_net_income in step 6, but never as a scored cell.

What the two models predict

Stylized predictions designed to make the trade-off visible. The tax-focused model nails taxes but zeros out benefits and flags everyone as Medicaid-ineligible; the benefit-focused model gets the eligibility calls right but is rough on tax amounts. Predictions are LLM outputs — they include the boolean for Medicaid but never the dollar value.

How booleans fit in

Eligibility flags like medicaid_eligible are a different prediction task from a dollar amount. We handle them in the same single-bucket scheme by separating two concerns:

  • Score the LLM only on the boolean (true or false). It never has to estimate the per-capita value.
  • Weight the boolean using PolicyEngine's paired value variable (medicaid_value), so a wrong eligibility call for someone with $9,000 of Medicaid value counts more than the same wrong call with $0 at stake.

All the weighting schemes below (equal, per-household impact, global, budget, floor) use this paired value as |ref| for the boolean. Row scores stay binary for booleans — wrong eligibility = 0, right eligibility = 1.

2Row score

The atomic unit. We need this before we can talk about weighting.

For one household and one output, the row score compares the prediction to the reference and rescales the error. The reference-zero case gets its own line so boolean outputs and dollar zeros are handled identically: any nonzero prediction on a zero reference scores 0.

// when the reference is zero (no liability, no benefit, or eligibility = false)
row_score = 1 if pred = 0 else 0

// otherwise
row_score = max(0, 1 − |pred − ref| / |ref|)

No `$1,000` floor in the denominator: the max(0, …) already caps the score at 0 when the error exceeds the reference, so the floor wasn't doing any extra work. Bonus: booleans (ref ∈ {0, 1}) drop out of this same formula at 0 or 1 with no special branch needed.

3Equal weights

The simplest combination: each output counts the same inside each household.

Average the five row scores into a household score, then average the household scores into a model score. Each output gets a weight of 1/5 in every household.

Transparent Treats $20 like $4,000 Benefit-heavy mix wins by default

Because most outputs are benefits or near-zero for most households, a model that just outputs zeros and "not eligible" everywhere does well — it gets full credit on every correct zero. Predicting the actual dollar amount of a benefit, or the correct Medicaid eligibility, is worth no more than predicting that it is zero when it actually is zero.

4Per-household impact weights

Inside each household, weight each output by its share of total absolute dollars.

impact_weight_ij = |ref_ij| / Σ_k |ref_ik|

Now an output gets weight in proportion to its dollar size for that household. A $5,000 income tax line carries more weight than a $500 ACA premium tax credit.

Rewards big-dollar accuracy A household with one nonzero output gives that output 100% weight Models can game per-household quirks

The failure mode: if a household's only nonzero output is income tax, getting income tax right earns 100% of that household's score. Predicting zeros for the others doesn't matter — they had zero weight anyway. A model can score well by being good at the dominant output for each household without being good in general.

5The 30% floor (the current bandaid)

Blend equal weights and household-impact weights with a floor of 30% equal weight per output.

weight_ij = 0.3 × (1/K) + 0.7 × impact_weight_ij

This patches the per-household failure mode by guaranteeing every output some minimum weight. But the 30% is a free parameter — change it to 25% or 40% and the leaderboard shifts. Nothing in the data tells us which value to use.

Caps the per-household failure 0.3 is arbitrary — it picks the winner Two hidden assumptions stacked together

6Global variable weights Recommended

Take the per-household impact share, average it across households, and use that as one fixed weight per variable.

// Step 1: per-household denominator — whichever is larger between
// net income and the gross tax-benefit flow.
net_income_i = market_income_i + Σ_k signed_contribution_ik
denom_i = max(|net_income_i|, Σ_k |ref_ik|)

// Step 2: per-household share for each variable
share_ij = |ref_ij| / denom_i

// Step 3: mean across households, then renormalize so weights sum to 1
raw_j = mean_i share_ij
weight_j = raw_j / Σ_k raw_k

// Step 4: score each household with the same weights, then average
household_score_i = Σ_j weight_j × row_score_ij
model_score = mean_i household_score_i

The denominator is the key move. A naïve "share of total household activity" using Σ |ref| alone would give a $1 benefit to a high-earning household 100% weight in that household (it's the only thing in the numerator). The bounded denominator max(|net income|, Σ |ref|) fixes this: when the household has substantial market income, net income is large and a $1 program becomes 1 / net_income — correctly dwarfed. When taxes and benefits cancel each other out and net income drops below the gross flow, the denominator falls back to Σ |ref| so shares never sum past 1.

Booleans like Medicaid eligibility carry weight through both ends: their |ref| is PolicyEngine's paired per-capita value, and that same value contributes to net income when eligibility is true. The LLM is still graded only on the boolean call itself.

No arbitrary floor Stable under rich-slice-tiny-benefit Bounded — per-household shares ≤ 1 Derived from the benchmark, not chosen

7Two-score breakdown

The methodologically cleanest option: report two numbers instead of forcing one.

Amount accuracy scores only the dollar cells where the reference is nonzero, using global variable weights from step 6 (booleans don't have a magnitude to predict, so they're excluded). Participation accuracy scores the zero-vs-positive call on dollar cells and the eligibility call on boolean cells — both binary checks on every cell.

amount_score = mean_i Σ_{j: amount, ref_ij ≠ 0} weight_j × row_score_ij / Σ_{j: amount, ref_ij ≠ 0} weight_j

participation_score = fraction of cells where the binary call matches:
  // amount cells (pred_ij = 0) matches (ref_ij = 0)
  // boolean cells pred_ij = ref_ij
Cleanest separation Two numbers, not a leaderboard

Useful as a companion to the global-weights score: it shows whether a model is failing on dollar amounts or on whether a benefit applies at all. If a single leaderboard column is non-negotiable, the next section walks through the precise math for collapsing A and P into one number.

8Collapsing A and P into one number

If you decide a single leaderboard column is non-negotiable, here is the precise math. Five options, each with the formula and what it does to the worked example.

Option 0: skip the combine

The global variable weights score from step 6 already counts every cell, including correct zeros, so participation is already inside it. Use that as the headline and treat amount and participation as a diagnostic split rather than components of the combined score. Cheapest defense for reviewers, no new combine function to argue about.

Option 1: arithmetic mean

S = (A + P) / 2

Easy to read. Lets a model compensate (e.g. 100% on participation, 0% on amount → 50% combined), which reviewers will flag as papering over a real failure.

Option 2: geometric mean

S = √(A · P)

Zero on either component zeros the combined score. Penalizes imbalance smoothly. Common in compound rates, reasonably defensible.

Option 3: harmonic mean (F1-style)

S = 2 · A · P / (A + P)

The convention for combining two complementary 0-to-1 metrics. Reviewers already recognize the F1 pattern, so this needs the least defense. Penalizes imbalance more than arithmetic but less than min.

Option 4: minimum

S = min(A, P)

A model wins only if its weakest dimension is best. Strict but easy to justify: "good at both, evidenced by the worse of the two." Tends to flip rankings versus the means because tied weak sides are common.

Option 5: weighted (tunable)

S = α · A + (1 − α) · P // or generalized mean of order p

Brings back the same free knob as the 30% floor unless α is derived from the data (e.g. the fraction of cells where the reference is nonzero). Avoid as a primary metric; reviewers will press on the choice of α.

How they rank the two models

Arithmetic / geometric / harmonic agree directionally Min flips the winner

Arithmetic, geometric, and harmonic all rank benefit-focused above tax-focused because they reward its big lead on participation. Min flips it: tax-focused's weak side is participation, benefit-focused's weak side is amount, and tax-focused's weak side is the higher of the two — so it wins the strict "good at both" test.

Recommendation: ship the global variable weights score (option 0) as the headline. If a column has to be amount-and-participation combined, use the harmonic mean — the only one in this list that reviewers won't ask you to justify from first principles.

9Budget-weighted variable weights

Same shape as the recommended metric, but weights come from aggregate dollar volume across all households instead of from averaging household shares.

weight_j = Σ_i |ref_ij| / Σ_i,k |ref_ik|

Each variable's weight is the share of all reference dollars it accounts for in the benchmark. Big aggregate programs dominate; small ones recede. This is what you would use if you wanted to score against fiscal impact rather than household impact.

Data-derived, no free knob Big programs dominate Households with bigger totals pull more weight

Versus global variable weights: budget tilts even further toward Medicaid (the $9,000 value in H3 dominates the aggregate) and less toward income tax (medium for everyone, small in aggregate). The household-perspective question — how well does this model do for an average household — bends toward fiscal aggregate accounting instead.

10Net-income reconstruction

Different paradigm. Skip per-output weights and score the summed household bottom line instead.

net_i = Σ_j ∈ amount value_ij + Σ_j ∈ binary eligible_ij · paired_value_ij

score = mean_i row_score(net_ref_i, net_pred_i)

Adds up all components for the household: dollar amounts go in directly, and each boolean contributes its paired per-capita value when the LLM predicts "eligible" (zero otherwise). Scores how close the predicted bottom line is to the reference. No component-weighting choice because the components are added rather than averaged.

No component weighting choice Maps to household well-being Offsetting errors hide mistakes Components can't be diagnosed

Failure mode: a model that overstates taxes by $1,000 and overstates benefits by $1,000 scores perfectly on net even though both components are wrong. Useful as a sanity check alongside a component-level metric, not as a replacement.

11Other variants we considered

Defensible but not chosen, with the reason in each card.

Geometric mean across cells

Replace the arithmetic mean of row scores with a geometric mean. A single zero score drags the household score to zero. Penalizes any catastrophic miss instead of letting good outputs average them out. Hard to recover from one wrong cell, which is too punishing for a benchmark across many outputs.

Median household instead of mean

Robust to outlier households. With three households it makes no statistical sense; with thousands it would dampen the influence of unusual cases. Worth keeping in mind for the full benchmark but not the primary metric — we want models that work for everyone, not just the typical household.

Stratified by household type

Group households by income decile or family structure, score within each stratum, then average across strata. Guarantees every household type is represented equally, regardless of how the sample was drawn. Worth doing as a report breakdown, but stratum choice is its own free parameter.

Threshold accuracy (got it / didn't)

Score 1 if the prediction is within X% of the reference and 0 otherwise. Conceptually clean and easy to communicate, but the threshold X is the same arbitrary knob as the 30% floor — in a different costume.

Smooth continuous row score

Replace max(0, 1 − err/scale) with 1 / (1 + err/scale) or exp(−err/scale) so the score never clips at zero. Lets the metric distinguish between "very wrong" and "extremely wrong." Useful for diagnosing model behavior on hard rows, but harder to read as a percentage.

12Side by side

The same two models, scored every way. Different choices give different winners.

Recommendation: ship bounded global variable weights (denominator = max(|net income|, Σ |ref|)) as the default headline ranking, paired with the amount + participation breakdown as diagnostics. If a single combined column is needed, use the harmonic mean of amount and participation. Drop the 30% floor — it's the only option whose result moves when you change a number that isn't in the data.

Offer equal weights and budget-weighted as opt-in alternatives in the app and paper. Both are defensible and worth showing next to the headline so readers can see how the ranking shifts under different stated philosophies — one household = one household, versus one dollar = one dollar. Keep net-income reconstruction available as a sanity check.
